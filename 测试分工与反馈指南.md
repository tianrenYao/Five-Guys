# 测试分工与反馈指南

本文档面向项目 4 位组员（测试员 A / B / C / D），目标是用一周时间完成功能验证，并产出**专业、可执行**的反馈与改进建议。

> **核心原则**：**优先完善已有功能**，避免无脑加新功能；任何新功能建议必须能回答"它服务于哪个核心场景（碳追踪 / 废物管理 / 报告生成 / 告警）"，否则降级为 P2。

| **B** | **姚天任** | 报告与告警 — Report + AI + PDF + Alert | 判断生成质量、AI 输出、告警合理性 | 判断密集 | 抽测 C 的 Map |

## 3. 各测试员任务详细清单

### 🅰 测试员 A — 尤浩洋 — 数据录入（Carbon / Waste / OCR）

**功能清单**：

- 新增 / 编辑 / 删除碳排放记录、废弃物记录
- 不同 category 的下拉、计量单位换算
- 上传水电费 PDF/图片 → OCR 自动识别 → 自动填表
- 导入 / 导出 CSV（如果有）
- 表单校验：负数、超大数字、空值、特殊字符、未来日期

**重点踩坑场景**：

- 上传一张**很糊的截图**或**不是账单**的图，看是否友好报错
- 同一日期同一门店重复录入是否被允许 / 提醒
- 改完记录后 Dashboard 数字是否实时更新

**优先建议方向**：

- 录入流程是不是繁琐？能不能合并字段、加默认值？
- OCR 识别错的字段如何快速修正？
- 列表页大数据时排序/筛选/分页是否流畅？

### 🅱 测试员 B — 姚天任 — 报告与告警（Report / AI / PDF / Alert）

**功能清单**：

- 生成各种类型的报告（monthly / quarterly / annual / custom）
- AI ESG Analysis 按钮（已修复，markdown 渲染）
- PDF 导出（已修复，已经能用）
- Alert 阈值设置、告警日志、未读计数、邮件通知
- 删除报告 / 重新生成 AI 分析

**重点踩坑场景**：

- 生成一份**没有数据的时间段**报告（比如未来），内容是否合理
- AI 反复点 "Re-generate" 看是否会报错或重复扣 token
- PDF 中文字符是否完整
- 阈值设得**很低**触发告警，看邮件是否真送达
- 告警日志的"已读"状态是否持久化

**优先建议方向**：

- AI 输出格式现在好不好看？还有哪些 markdown 元素没渲染好？
- PDF 排版是否专业？哪些图表/章节缺了？
- 告警是否会"狼来了"——阈值默认值合不合理？

### 🅲 测试员 C — 李思航 — 可视化与洞察（Dashboard / Map / Anomaly / Compliance）

**功能清单**：

- Dashboard KPI 卡片、趋势图、门店列表
- Map 上的门店标记、点击弹窗、热力图（如果有）
- Anomaly Detection 异常数据识别（统计离群点）
- Compliance 法规符合度跟踪

**重点踩坑场景**：

- Dashboard 在**完全没数据**的新公司账号下是否崩溃 / 显示空状态
- Map 在地图无法加载（网络问题、API key）时如何降级
- 异常检测的判定标准是什么，会不会过敏 / 过钝
- 切换不同区域 / 时间段，所有图表是否都刷新

**优先建议方向**：

- 数据多的时候图表是不是看不清？有没有需要的对比/筛选？
- Map 是不是只显示信息但没有行动建议？
- 异常检测发现问题后**下一步**怎么办？流程是否完整？

### 🅳 测试员 D — 卢禹安 — 治理与管理（Auth / Admin / Audit / Supplier / Policy / Training）

**功能清单**：

- 登录、登出、记住我（如果有）、密码错误锁定
- Admin 创建 / 修改 / 禁用用户、分配角色
- Audit Log 记录所有写操作、可搜索可筛选
- Supplier 上下游供应商管理
- Policy 内部 ESG 政策文档
- Training 员工培训记录

**重点踩坑场景**：

- 用 `business` 角色去访问 `/admin/users` 应该被拒绝（403/重定向）
- 用 `region_manager` 去操作非自己区域的门店数据应该被拒绝
- Audit Log 里能不能搜到自己刚才的操作？时间戳对吗？
- 删除一个被引用的供应商，是软删还是连带删？
- 同一个邮箱注册两次的处理

**优先建议方向**：

- 权限粒度够不够细？区域经理能否被精确到只看 X 个门店？
- Audit Log 太杂还是太空？哪些操作其实没必要记？
- Supplier / Policy / Training 这三个模块的**真实使用流程**是否合理？

## 4. 反馈模板（必须用，不然意见杂乱无章）

所有 Bug / 改进建议都按下面两份模板，一条一张卡，提交到共享文档（飞书 / 腾讯文档 / Notion 表格皆可）。

### 🐛 Bug 报告模板

```markdown
**模块**：Report - PDF 导出
**严重度**：⬜阻塞 ⬜严重 ⬜一般 ⬜轻微
**重现步骤**：
1. 用 hq_manager 登录
2. 生成一份月度报告
3. 点击 Export PDF
**期望结果**：浏览器下载 PDF
**实际结果**：500 错误
**截图/日志**：（贴图或贴 console error）
**测试环境**：Chrome 146 / macOS / http://csi6220-1-vm2.ucd.ie
```

### 💡 改进建议模板

```markdown
**模块**：Carbon - 数据录入
**类型**：⬜UX 优化 ⬜性能 ⬜功能完善 ⬜新增功能
**问题描述**：录入电费时需要先选 category 再选 unit，繁琐
**建议方案**：选择"electricity"后自动锁定 unit 为 kWh
**理由**：减少 1 次点击，避免选错单位
**优先级**：⬜P0 必须 ⬜P1 强烈建议 ⬜P2 锦上添花
```

> **不要自己费力写 Bug 报告**——直接用第 5、6 节的"魔法提示词"让 AI 帮你写。

## 8. 开工说明（请所有人先看完再动手）

1. 你的模块、你的角色账号在第 2、3 节
2. 提交 Bug / 改进 → 共享文档里的卡片，使用第 4 节模板
3. **不要自己费力写 Bug 报告**：复制第 6 节的提示词 A / B 让 AI 帮你写
4. 开新 AI 对话**务必先粘第 5 节的项目上下文**，否则 AI 输出会跑偏
5. 看到红色错误？复制错误文本 → 粘进提示词 D 让 AI 解读
6. 严格遵守：**优先完善已有，避免乱加新功能**

> 目标：花 30% 时间发现问题，70% 时间靠 AI 整理输出，**完全不靠自己挤词**。
